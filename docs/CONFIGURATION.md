# Configuration reference — what to fill in, where, and where it comes from

Every parameter you set, grouped by the file it lives in, with **where to source
the value**. Read [PREREQUISITES.md](PREREQUISITES.md) first (tools + AWS login);
run with [DEPLOYMENT.md](DEPLOYMENT.md).

## The files at a glance

| File | Workspace / phase | You edit it | Contains |
|------|-------------------|-------------|----------|
| `terraform.tfvars` | root (Phases 1a/1b/3/R2) | **yes** — copy from `.example` | regions, CIDRs, Panorama sizing, FW license/PIN, key pair, DC, R2/GA flags |
| `phase2-panorama-config/terraform.tfvars` | panos workspace (Phase 2a/GP/EDL/log-collector) | **yes** — copy from `.example` | Panorama password, object names, dataplane IPs, GP + EDL + log-collector toggles |
| `panorama_vm_auth_key.auto.tfvars` | root (auto-loaded) | **no** — generated by Phase 2a | the `panorama_vm_auth_key` handoff |
| `optional/eks-deploy/terraform.tfvars` | EKS add-on | only if you deploy EKS | IDs from root outputs + WordPress/EKS settings |

> Secrets never leave your machine: `*.tfvars`, `*.pem`, `*.key`,
> `*.auto.tfvars` are git-ignored. Only `*.tfvars.example` is committed.

## Manual vs auto-derived — the one distinction that matters

Every value is either something **you** provide, or something the deploy
**derives** for you. Don't hand-set the auto-derived ones — copy them from
`terraform output` when a later phase asks for them.

| Kind | What | Examples |
|------|------|----------|
| **Manual — external** (only you can get these) | PANW CSP + AWS account inputs | `fw_auth_code`, `fw_registration_pin_id`/`_value`, `panorama_serial_number`, `panorama_device_otp`, `key_name`/`ssh_public_key`, Marketplace subscription |
| **Manual — you choose** | passwords, names, sizing, feature flags | `panorama_admin_password` (= phase2 `panorama_password`), `dc_safe_mode_password`, `gp_local_users`, CIDR plan, `enable_region_b`/`enable_global_accelerator`/`enable_custom_domain`, `custom_domain_subdomain_zone` |
| **Auto — read from `terraform output`** | IPs, FQDNs, EIPs, allocation IDs, GA anycast IPs, custom-domain NS + cert PEMs | `fw_public_eips`, `panorama_private_ip`, `fw_mgmt_private_ips`, `global_accelerator_static_ips`, `custom_domain_name_servers`, `custom_domain_gateway_fqdns`, `custom_domain_cert_pem`/`_key_pem` |
| **Auto — cross-workspace handoff** | written to a file, loaded by the next phase | `panorama_vm_auth_key` → `panorama_vm_auth_key.auto.tfvars` (never set by hand) |
| **Auto — derived in code** | subnet CIDRs, per-device template variables, serials-by-name overrides | subnet `.1` gateways, `$fw_untrust_ip`/`$fw_untrust_floating`, floating `.100/32` |

The one **manual step that involves an external system** beyond CSP/AWS is the
custom-domain **NS delegation** (parent domain → the Route53 name servers in
`terraform output custom_domain_name_servers`) — see
[the custom-domain walkthrough](DEPLOYMENT.md#custom-domain-optional--branded-browser-trusted-portal).

## A. Root — `terraform.tfvars`

Copy the template and edit: `cp terraform.tfvars.example terraform.tfvars`.

### Must set (deploy fails or is useless without these)

| Variable | Where to get it | Example |
|----------|-----------------|---------|
| `fw_auth_code` | **PANW CSP Portal** → Products/Assets → **Auth Codes** (VM-Series BYOL, with credit balance) | `"D1234567"` |
| `fw_registration_pin_id` | CSP Portal → **Assets → Device Certificates → Generate Registration PIN** (save the **ID**) | `"1A2B3C..."` |
| `fw_registration_pin_value` | same PIN generation (save the **value**); one PIN works for both FWs | `"XyZ..."` |
| `key_name` | the **NAME** of an EC2 key pair in the region ([PREREQUISITES §7](PREREQUISITES.md#7-ssh-key-pair)) — **not a file path.** The matching local `.pem` path goes in the separate `ssh_private_key_file` (default `~/.ssh/<key_name>.pem`). A `plan`-time validation rejects `key_name` values containing `/` or ending in `.pem`. | `"awsha-key"` |
| `panorama_admin_password` | **you choose** — Panorama admin password. Phase 1a sets it on Panorama automatically over SSH (needs the `key_name` private key at `~/.ssh/<key_name>.pem` or `ssh_private_key_file`). Use the **same** value as phase2 `panorama_password`. | `"<choose a strong password>"` |
| `dc_safe_mode_password` | **you choose** — the DSRM password for AD DS (only needed if `dc_promote_to_dc=true`) | `"Dsrm@Passw0rd!"` |

## Where the PANW values come from (CSP Portal)

1. **Auth code** (`fw_auth_code`): comes from a **Software NGFW Credits
   Deployment Profile** (CSP Portal → **Products → Software NGFW Credits →
   Create Deployment Profile**). The profile *is* what mints the VM-Series
   auth code; its size caps how many FWs/vCPUs you can license from the credit
   pool. Check the **credit balance** — credits deplete per redeploy
   (`NOV-021 Insufficient credits` if empty). Size it as:

   | Deployment Profile field | Region A only | Full multi-region (A + B) | Notes |
   |---|---|---|---|
   | **Number of firewalls** | **2** | **4** | Active/Passive HA pair per region — **both members consume a license** (the passive one too). |
   | **Planned vCPU / firewall** | **4** (repo default) — **8** for headroom | same | Must match `fw_instance_type`. Repo default is now `m5.xlarge` = **4 vCPU** (fits the 4 required ENIs, fewer flex credits). Bump to `m5.2xlarge` = **8 vCPU** for GP SSL/IPsec + inspection throughput. See the sizing note below. |
   | **Firewall model** | Flexible vCPU (VM-Series) | same | Use flexible vCPU, not a legacy fixed model. |
   | **Security use case / subscriptions** | must include **GlobalProtect** (the primary deliverable) + Advanced Threat Prevention + Advanced URL Filtering + WildFire + DNS Security | same | For a lab, enable the full bundle. GP is non-negotiable here. |

   **Panorama is licensed separately** (not from the per-vCPU VM-Series flex
   count): a Panorama management license sized for the number of managed
   devices (**2** for Region A, **4** for A+B) plus log storage (Strata Logging
   Service or on-box) at a modest lab volume. Panorama's own `m5.4xlarge`
   (16 vCPU) is an EC2 sizing, **not** a flex-vCPU license input.

   > Don't over-provision the firewall count "for headroom" — credits deplete
   > per redeploy, so a too-large profile just drains the pool faster. Size to
   > what you run concurrently (2 now; 4 once Region B is live).
   >
   > **Why 4 vCPU/FW by default, and when 8?** The hard sizing constraint on
   > AWS is the **ENI count**, not vCPU: each FW needs **4 ENIs** (mgmt /
   > untrust / trust / ha2). `m5.xlarge` (4 vCPU) and `m5.2xlarge` (8 vCPU)
   > *both* cap at **4 ENIs** (verified via `describe-instance-types`), so 4
   > ENIs fit either way. The repo default is now **`m5.xlarge` (4 vCPU /
   > 16 GiB)** — a supported VM-Series tier (min memory for 4 vCPU ≈ 9 GB, well
   > under 16), handles the required 4 ENIs, and consumes fewer flex credits per
   > FW: the sensible lab choice. Bump to **`m5.2xlarge` (8 vCPU / 32 GiB)** for
   > more dataplane cores when GlobalProtect SSL/IPsec termination + threat
   > inspection throughput matters — a performance choice, not a requirement.
   > Whichever you pick, `fw_instance_type` **and** the profile's vCPU/FW must
   > agree (4 ↔ `m5.xlarge`, 8 ↔ `m5.2xlarge`).
2. **Registration PIN** (`fw_registration_pin_id` + `_value`): CSP Portal →
   Assets → **Device Certificates → Generate Registration PIN**. Note both the ID
   and value; pick the shortest expiry that covers your deploy window (it expires).
3. **Marketplace subscription** (not a variable): subscribe to the VM-Series and
   Panorama **BYOL** AMIs once per AWS account (console) — verify with
   `scripts/accept-marketplace-terms.sh`.

## Certificates for GlobalProtect

`gp_server_cert_pem` / `gp_server_key_pem` are the TLS server cert + private key
for the portal/gateway FQDN. **Where they're used downstream:** both go into a
single `panos_ssl_tls_service_profile` (`modules/panorama_config/gp.tf`), which
is attached to **both** the GP Portal and the GP Gateway — in this single-region
(Region A) setup they share the same FW public EIP/FQDN, so **one cert covers
both**. Once Region B (Phase R2) is added, check whether each entry in
`gp_external_gateways` uses its **own** FQDN (`address`) — if so, each needs its
**own** matching cert (a single cert's CN/SAN can't validate for two different
hostnames unless it's a wildcard or multi-SAN cert covering all of them).

**Lab/self-signed — concrete, copy-pasteable path** (recommended default for
this project; clients will show a trust warning, which is expected and fine
for testing):

```bash
# CN/SAN MUST match the address your GP client will actually connect to —
# usually the Region A FW's public EIP itself (no DNS needed for a lab), or a
# DNS name you point at it. Get the EIP first:
terraform output fw_public_eips   # {"region_a": "<eip>", ...}

openssl req -x509 -newkey rsa:2048 -keyout gp-key.pem -out gp-cert.pem \
  -days 365 -nodes -subj "/CN=<fw-public-eip-or-your-dns-name>"
```

Paste the two PEMs into `phase2-panorama-config/terraform.tfvars` as **heredocs**.
`.tfvars` files can't call functions — `file()`/`path.module` are rejected there
("Functions may not be called here") — so inline the content with `<<-EOT`:

```hcl
gp_server_cert_pem = <<-EOT
-----BEGIN CERTIFICATE-----
...paste gp-cert.pem verbatim (whole chain if you have one)...
-----END CERTIFICATE-----
EOT
gp_server_key_pem = <<-EOT
-----BEGIN PRIVATE KEY-----
...paste gp-key.pem verbatim...
-----END PRIVATE KEY-----
EOT
```

(If you prefer `file()`, put the assignment in a real `.tf` file, not `.tfvars`.)
`terraform.tfvars` is git-ignored, so the inlined key stays out of git.

**Other options**, if you need clients to trust the cert without a warning:
- **Automated Let's Encrypt (recommended for a real domain):** turn on the
  `custom_domain` module (`enable_custom_domain=true`, `cert_mode="letsencrypt"`).
  It issues a **wildcard** cert via DNS-01 against the Route53 subdomain it
  creates, and exposes it as `terraform output custom_domain_cert_pem` /
  `_cert_key_pem` — paste those into `gp_server_cert_pem`/`gp_server_key_pem`.
  One wildcard covers the portal + all gateway FQDNs. Full walkthrough:
  [DEPLOYMENT.md](DEPLOYMENT.md#custom-domain-optional--branded-browser-trusted-portal).
- **Public CA** (manual): issue a cert for your real DNS name (e.g.
  `vpn.example.com`), paste the PEM cert + private key the same way.
- **ACM**: request/import in ACM, then **export** it (ACM private CA export) —
  PAN-OS cannot reference an ACM ARN directly; the exported PEM must live on
  the FW, same as the self-signed/public-CA paths above.
